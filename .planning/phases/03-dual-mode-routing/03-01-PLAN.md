# Phase 03 Plan 01: Routing Foundation

**Duration estimate:** 1–2 hours
**Risk level:** Medium — refactors 4 existing components + the base layout; wrong test coverage could break the Planning home

## Goal

Migrate the dual-mode system from "same page, CSS flip" to "real routes, View Transitions animate the swap." By the end of this plan, clicking the mode toggle anywhere on the Planning home navigates to `/onsite`, the URL changes, the palette flips smoothly, the back button returns correctly, and the existing Planning home experience is preserved unchanged.

The `/onsite` route itself is a stub — just enough to prove routing works. Actual landing page content comes in 03-02.

## Success Criteria (what must be TRUE by the end)

1. Opening `http://localhost:4321/` shows the Planning home exactly as it does today (no regressions)
2. Clicking the "At Kunsthaus" segment in any of the 4 mode toggles navigates to `http://localhost:4321/onsite` with a smooth View Transitions animation (not a hard reload)
3. Clicking the "Plan a Visit" segment on `/onsite` navigates back to `/` with the same animation
4. `html[data-mode]` is set to `"planning"` on `/` and `"onsite"` on `/onsite`
5. The semantic token cascade applies the dark palette on `/onsite` and the light palette on `/`
6. Each mode toggle's active segment reflects the current pathname (start of `/onsite` → onsite segment active; else planning segment active)
7. localStorage no longer stores `kunsthaus-mode` — check: `localStorage.getItem('kunsthaus-mode')` returns `null` on a fresh session
8. Browser back/forward buttons navigate between `/` and `/onsite` correctly
9. `npm run build` succeeds with no errors
10. `npm run dev` runs without console errors

## Anti-Patterns to Avoid

1. **Don't use client-side navigation via `window.location.href`** — use `<a href>` tags so Astro's View Transitions router intercepts the navigation naturally.
2. **Don't hydrate active segment state from localStorage** — the server-rendered HTML should already have the correct active state based on `Astro.url.pathname`. Client-side hydration only needs to update state after `astro:page-load` events.
3. **Don't leave dead localStorage keys** — remove ALL reads and writes to `kunsthaus-mode`. Don't just comment them out.
4. **Don't set `data-mode` via JS in the new flow** — the page's layout sets it at server render time. JS setting of `data-mode` was the old pattern and is being retired.
5. **Don't duplicate the pathname-matching logic across all 4 writers** — extract a shared utility function in `src/lib/mode.ts` or similar if it starts getting copy-pasted. Simple one-liner is fine: `const isOnsite = Astro.url.pathname.startsWith('/onsite');`
6. **Don't forget the overlay-close-before-navigate step** in HamburgerOverlay — without it, the overlay stays visible during the View Transitions animation and looks weird.

## Notes

- **View Transitions browser support:** Chrome 111+, Safari 18+, Firefox behind a flag. For unsupported browsers, navigation happens as a normal page load (graceful fallback). Don't worry about this for the pitch demo — most modern browsers support it.
- **Astro-specific events:** `astro:page-load` fires after a View Transitions navigation completes. Use this instead of `DOMContentLoaded` for any client-side logic that needs to re-run on each navigation.
- **Pill animation compatibility:** The existing pill-slide logic uses `gsap.to()` to animate the pill position when a segment gains the `is-active` class. After this refactor, the `is-active` class is set by server-rendered HTML (no animation needed on first paint) and by client-side route detection (animation fires on route change). Both paths need to work.
- **No changes to `/`'s visual appearance:** This plan is a pure refactor of the state layer. If `/` looks different after this plan, something is wrong.
