## Critical Anti-Patterns (do NOT repeat these)

| Pattern | Description | Severity | Prevention |
|---------|-------------|----------|------------|
| Using `window.location.href` for mode navigation | Bypasses Astro View Transitions, causes full page reload | advisory | Use `<a href>` tags so Astro intercepts them naturally |
| Setting `data-mode` via JS | Causes state divergence from the URL (the new source of truth) and breaks the layout-prop pattern Task 1 established | advisory | `data-mode` is set only by `BaseLayout` from the `mode` prop. Delete any `document.documentElement.setAttribute('data-mode', ...)` calls found in Tasks 3-6. |
| Leaving dead `kunsthaus-mode` localStorage code | Dead code that suggests the old pattern is still in use | advisory | Remove ALL reads and writes to `kunsthaus-mode` in Tasks 3-6. Run `grep -rn 'kunsthaus-mode' src/` at the end of Task 7 — must return zero matches outside comments. |
| Hydrating active segment state from localStorage | The server-rendered HTML already has correct active state based on `Astro.url.pathname`; client hydration should only update state on `astro:page-load` (after navigation completes) | advisory | Server-render `is-active` + `aria-checked` from `Astro.url.pathname.startsWith('/onsite')`. Client only reacts to `astro:page-load` events. |
| Forgetting overlay-close-before-navigate in HamburgerOverlay | Without it, the overlay stays visible during the View Transitions animation and looks weird | advisory | Task 4 has a specific note on this. Add a click handler on the mode segment anchors that calls `close()` on the overlay. |
